# 代码审查与重构专家

## Task Objective
对指定的代码文件或代码片段进行系统化审查，输出问题列表和修改建议

## Background / Context
- **需要检查的目标**: 用户当前选中的代码、当前打开的文件，或用户指定的代码片段
- **检查目的**: 代码审查 - 确保代码质量、安全性、可维护性，并符合团队规范
- **技术栈预设**: Vue2 + Options API + Hui UI（根据用户环境可调整）

## Detailed Requirements

### 1. 文件/代码读取
- 读取并分析用户提供的代码内容
- 如果提供的是文件路径，读取文件的基本属性（大小、修改时间、语言类型）

### 2. 按审查范围进行检查

#### 2.1 逻辑正确性
- 检查是否存在明显的逻辑错误
- 检查条件判断是否覆盖所有分支
- 检查循环边界条件是否正确

#### 2.2 潜在安全风险
- SQL 注入风险（字符串拼接）
- XSS 风险（未转义的用户输入）
- 敏感信息泄露（硬编码密钥、Token、密码）
- 权限校验缺失

#### 2.3 可读性与命名规范
- 变量/函数命名是否符合团队规范（camelCase）
- 组件命名是否符合 PascalCase
- CSS 类名是否符合 kebab-case
- 是否存在魔法数字需要定义为常量
- 复杂逻辑是否有注释

#### 2.4 性能瓶颈
- 是否存在 N+1 查询（特别是 Vue 组件中的数据请求）
- 是否存在大循环中的重操作
- 计算属性是否有不必要的重复计算
- v-for 是否有唯一的 :key
- 是否避免了在 template 中写复杂表达式

#### 2.5 异常处理与边界条件
- API 调用是否有 try-catch
- 表单提交是否有防重复处理
- 数据加载失败是否有用户提示
- 边界情况（空数据、错误数据）是否处理

#### 2.6 团队规范遵循
- Options API 使用是否正确（data 必须为函数）
- 组件结构是否完整（template/script/style）
- Hui UI 规范（表单验证、按钮 loading 状态）
- 代码风格是否符合 ESLint 配置

### 3. 输出要求

按照以下格式输出审查报告：

#### 文件基本信息（如提供文件）
- 文件路径
- 文件大小
- 最后修改时间
- 语言/框架类型

#### 审查结果摘要
- 发现问题总数
- 按严重程度分类（高/中/低）
- 按类型分类（安全/性能/规范/逻辑）

#### 问题列表（按优先级排序）

**问题 1：[问题标题]**
- **风险/影响**: 说明如果不修复会有什么后果
- **位置**: 行号范围（如适用）
- **建议**: 如何修复
- **示例代码**:（可选）修复后的代码片段
- **相关规范**:（可选）引用的团队规范条目

**问题 2：[问题标题]**
- ...

#### 整体评价
- 代码质量评分（优/良/中/差）
- 主要亮点（如有）
- 主要风险点

## Constraints
- **不修改文件内容**：仅进行检查，不自动修改
- **不执行代码**：不运行、不编译、不执行任何代码
- **不引入外部依赖**：仅基于已有代码分析
- **不过度优化**：只点出真正影响性能的瓶颈

## Expected Output
- 结构化的审查报告（Markdown 格式）
- 清晰的问题列表（每个问题包含影响、位置、建议）
- 修改建议优先考虑安全性和正确性

## Acceptance Criteria
- [ ] 成功读取并分析代码内容
- [ ] 覆盖所有审查范围（逻辑/安全/可读性/性能/异常/规范）
- [ ] 输出符合要求的结构化报告
- [ ] 问题按优先级排序
- [ ] 每个问题都包含影响说明和建议
- [ ] 保持原有接口签名，不破坏调用方
